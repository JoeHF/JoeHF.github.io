ReentrantLock详解

发表于 2020-04-16 | 分类于 java

ReentrantLock是可重入锁，实现了lock接口。ReentrantLock是从api层面提供了加锁和释放锁的功能，而synchronized是在jvm层面实现的线程同步。ReentrantLock实现了公平锁和非公平锁两种方式。

可重入锁

可重入锁顾名思义就是如果申请锁的线程当前拥有锁就可以直接获得该锁。需要注意的是如果锁被获取了n次，那么只有当同样释放n次之后，该锁才算是完全释放成功

公平锁和非公平锁

ReentrantLock提供两种构造函数来分别构造一个公平锁和非公平锁。

公平锁每次获取到锁为同步队列中的第一个节点，保证请求资源时间上的绝对顺序，而非公平锁有可能刚释放锁的线程下次继续获取该锁，则有可能导致其他线程永远无法获取到锁，造成“饥饿”现象。

公平锁为了保证时间上的绝对顺序，需要频繁的上下文切换，而非公平锁会降低一定的上下文切换，降低性能开销。因此，ReentrantLock默认选择的是非公平锁，则是为了减少一部分上下文切换，保证了系统更大的吞吐量。

源码解析

非公平锁的实现

先看一下非公平版本的lock方法，实际上是调用的compareAndSetState(0,1)来获得锁。compareAndSetState是来自AbstractQueuedSynchronizer的方法，因为无论是NonfairSync还是FairSync都是继承自Sync，而Sync是ReentrantLock的抽象内部静态类，继承自AbstractQueuedSynchronizer。先不研究AbstractQueuedSynchronizer里的代码，从注释来看，compareAndSetState的作用是如果当前值为expect值相同，则将同步状态设置为update，并且返回true。如果当前值跟expect值不同则返回false，这个设置是原子性的操作，并且底层有volatile读写语义，所以当多个线程来竞争锁时只有一个线程能够获得该锁。

如果compareAndSetState返回true，则将exclusiveOwnerThread设置为当前线程。exclusiveOwnerThread是抽象类AbstractOwnableSynchronizer的字段，而AbstractQueuedSynchronizer是继承于AbstractOwnableSynchronizer的。如果返回false，则调用acquire(1)

独占式获取同步状态，如果获取失败则插入同步队列进行等待。这里会调用至少一次tryAcquire。

回过头看NonfairSync的tryAcquire方法

nonfairTryAcquire实现在Sync里

如果当前状态为0，则原子性地设置为1，如果成功返回true，如果当前状态为1则判断锁的线程是当前线程，那么也成功返回true，否则失败返回false.

公平锁的实现

公平锁的tryAcquire方法会更加复杂一点。如果当前锁的状态为0，则需要先判断hasQueuedPredecessors，如果hasQueuedPredecessors返回false则通过compareAndSetState尝试更新锁状态。

hasQueuedPredecessors是AbstractQueuedSynchronizer里的方法，这里不细讲，作用是判断是否有比当前线程等待时间更长的线程在等待这把锁，如果有的话当前线程就拿不到锁也就实现了公平锁的功能。
