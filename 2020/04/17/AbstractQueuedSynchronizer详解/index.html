AbstractQueuedSynchronizer详解

发表于 2020-04-17 | 分类于 java

之前讲ReentrantLock的时候提到了AQS，AQS全称是AbstractQueuedSynchronizer，它是实现同步组件的重要部分。同步组件的实现者（比如ReentrantLock）通过使用AQS提供的模板方法实现同步组件语义。AQS负责实现同步状态的管理，阻塞线程排队，等待通知等等底层实现。这篇文章归纳整理AQS提供的若干底层方法。

AbstractQueuedSynchronizer里面维护了一个双向链表来保存当前竞争同步状态的线程以及他们相应的状态，用内部静态类Node来表示每一个这样的节点

这里面prev, next, thread字段都非常直接，因为是双向链表，所以需要有一个指向前驱和后继的字段，thread表明当前是哪个线程在竞争同步状态。waitStatus有四种状态（SIGNAL, CANCELLED, CONDITION, PROPAGATE）

  1. SIGNAL（-1）：当后继节点被阻塞时，当当前节点release的时候需要唤醒后继节点
  2. CANCELLED（1）：当前节点因为超时或者中断被取消。节点如果进入这个状态后就不会再离开这个状态，一个线程如果是这种状态那么它就不会被阻塞
  3. CONDITION（-2）：当前节点在条件队列里，它不会被同步队列使用除非他从条件队列转移过来，转移过来之后他的状态将会变为0
  4. PROPAGATE（-3）：releaseShared操作需要转播到别的节点。
  5. 0：以上状态都不是

waitStatus的改变都是通过CAS操作来保证原子性。

AQS内部有head和tail field来指向这个列表的头结点和尾结点。需要注意的是head节点是lazy initialized，它的修改只能通过setHead来进行，另外如果head存在，它的waitStatus将会保证是非CANCELLED状态。tail节点也是lazy initialized，它的修改只能通过enq方法来加入新的等候节点。

AQS里还有一个volatile的类型为int的state节点来保存同步状态。

接下来看几个辅助方法的实现

enq方法比较有意思，它首先拿到尾部节点，如果尾部节点为null，则同步地设置一个新的头部节点，接着将新加的节点放在这个新的头部节点后面。这个就是所谓head节点的lazy initialization。

接下来看几种常见的用途，分别是独占式锁、共享式锁、可中断锁、超时等待锁等特性的实现

独占式锁

void acquire(int arg)

从注释可以看出acquire方法是独占式获得锁，会至少一次地调用tryAcquire，成功的话返回true，否则当前线程将被加入队列，周期性地被阻塞和解锁。tryAcquire由具体实现者来实现，比如ReentrantReadWriteLock里的Sync，ThreadPoolExecutor的Worker。如果没有成功，则先调用addWaiter再调用acquireQueued。

addWaiter新建一个节点，设置nextWaiter是Node.EXCLUSIVE表明是独占式节点。如果当前尾部节点不为空，则将新加入的节点放在队列的尾部。如果尾部节点还不存在则通过enq方法加入队列，然后返回新添加的节点。

acquireQueued寻找前驱节点，如果前驱节点是头部节点，表明没有线程在等待，并且tryAcquire成功获得了锁，那么设置当前节点为头结点，如果前驱还有节点在等待，则调用shouldParkAfterFailedAcquire

如果前驱节点的状态是signal，则返回true，表明当前节点应该阻塞，如果前驱节点是CANCELLED状态，则更新链表，指向第一个waitStatus <= 0的节点，返回false。如果是0或者PROPAGATE状态，则将前驱节点waitStatus设置为SIGNAL。返回false。返回false之后acquireQueued的for循环会一直试图寻找前驱节点，这里就是线程自旋阻塞。

如果返回ture表明可以阻塞则调用parkAndCheckInterrupt

parkAndCheckInterrupt阻塞当前线程，返回当前线程是否被中断。如果中途发生了中断，但是线程还是会自旋直到当前线程拿到同步状态之后才会中断。

void acquireInterruptibly(int arg)

看完非响应中断独占式acquire之后再快速看下可中断版本

可以看到基本上实现是一致的，除了非响应中断在中断发生后只是设置interrupted为true，而响应中断则是抛出了异常。

boolean tryAcquireNanos(int arg, long nanosTimeout)

增加超时检测之后会计算出还剩余的等待时间nanosTimeout，如果小于0，则直接返回false，同时在阻塞前还检查了剩余时间是否大于spinForTimeoutThreshold（1000 纳秒），只有大于1000纳秒时才会阻塞当前线程。

boolean release(int arg)

unparkSuccessor是从末尾往前找到第一个后继节点这里非常让人困惑，好在知乎达人已经在我们之前探索了问题的答案，不得不佩服知乎里的牛人。https://www.zhihu.com/question/50724462，copy知乎里的回答：

① 处将新结点 node 的 prev 引用指向当前的 t, 即 tail 结点. 然而, 由于 ①, ② 这两行代码的合在一起并非原子性的, 所以很有可能在设置 tail 时存在着竞争, 也即 tail 被其它线程更新过了. 所以要自旋操作, 即在死循环中操作, 直到成功为止. 自旋地 CAS volatile 变量是很经典的用法. 如果设置成功了, 那么 从 node.prev 执行完毕到正在用 CAS 设置 tail 时, tail 变量是没有被修改的, 所以如果 CAS成功, 那么 node.prev = t 一定是指向上一个 tail 的. 同样的, ②, ③ 合在一起也并非原子操作, 更重要的是, next field 的设置发生在 CAS 操作之后, 所以可能会存在 tail 已经更新, 但是 last tail 的 next field 还未设置完毕, 即它的 lastTail.next 为 null 这种情况. 因此如果此时访问该结点的 next 引用可能就会得到它在队尾, 不存在后继结点的”错觉”. 而我们总是能够通过从 tail 开始反向查找, 借助可靠的 prev 引用来定位到指定的结点. 简单总结一下, prev 引用的设置发生在 CAS之前, 因此如果 CAS 设置 tail 成功, 那么 prev 一定是正确地指向 last tail, 而 next 引用的设置发生在其后, 因而会存在一个 tail 更新成功, 但是 last tail 的 next 引用还未设置的尴尬时期. 所以我们说 prev 是可靠的, 而 next 有时会为 null, 但并不一定真的就没有后继结点.
