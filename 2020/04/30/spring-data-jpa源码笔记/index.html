spring-data-jpa源码笔记

发表于 2020-04-30 | 分类于 spring

spring-data-jpa的优点很多，比如继承Repository接口，在注解中书写JPQL语句即可访问数据库；支持方法名解析方式访问数据库；使用Predicate支持动态查询等，在此不一一列举了。这些都是使用spring-data-jpa中的种种优点，要想将之使用的更好更优雅，就要从spring-data-jpa的加载和运行机制进行探秘。

spring-data-jpa在运行时和springframework框架实现了无缝对接。在使用spring的@Repository注解生成Repository实例时，使用动态代理类的方式对Repository接口进行了实例化并放入spring容器中备用。

RepositoryFactoryBeanSupport

RepositoryFactoryBeanSupport总的来说是用来方便的通过spring配置的方法setup repository factories。RepositoryFactoryBeanSupport是一个FactoryBean，其getObject方法如下：

那这个repository又是在哪里初始化的呢？搜索代码发现是在afterPropertiesSet

因为RepositoryFactoryBeanSupport实现了InitializingBean，所以会被beanFactory在所有properties都被设置之后调用。还可以看到这里repository并不是马上创建的，如果enable了lazyInit，那么只有当用到这个bean的时候才会初始化repository。

repository的创建需要用到RepositoryFactorySupport这样一个factory的getRepository方法。factory的创建是通过createRepositoryFactory来实现，这是一个抽象方法，由子类来实现。TransactionalRepositoryFactoryBeanSupport就是这样一个子类并且实现了createRepositoryFactory的方法。TransactionalRepositoryFactoryBeanSupport的doCreateRepositoryFactory同样交由子类来完成。TransactionalRepositoryFactoryBeanSupport只是添加了两个后置处理器exceptionPostProcessor和txPostProcessor。TransactionalRepositoryFactoryBeanSupport的子类我们就找到了JpaRepositoryFactoryBean。

来看一下JpaRepositoryFactoryBean的doCreateRepositoryFactory

从名字来看JpaRepositoryFactoryBean的doCreateRepositoryFactory返回的正是JpaRepositoryFactory，少了一个bean，正好是从工厂bean得到bean的过程。JpaRepositoryFactory就是通用的repository factory了，它也没有进一步的子类了，它扩展自RepositoryFactorySupport，而且并没有实现getRepository方法，getRepository实现在它继承的抽象父类里面。

接下来看一下RepositoryFactorySupport的getRepository方法，这个方法是在RepositoryFactorySupport里的。

其中的getTargetRepository又是一个抽象方法。JpaRepositoryFactory里的getTargetRepository返回的就是一个JpaRepositoryImplementation实例，JpaRepositoryImplementation也是一个接口，它继承了JpaRepository。JpaRepositoryImplementation是JpaRepositoryFactory里的内部类。

在这个方法中可以看到，首先它获取了Repository的各种信息，包括metadata、customImplementation、information等， 然后使用了ProxyFactory，并且为这个ProxyFactory注册了几个Advice，最后通过getProxy方法创建了一个Repository的代理。 而真正起到魔法般作用的应该就是这个QueryExecutorMethodInterceptor类了。

总结

RepositoryFactoryBeanSupport的作用是，为spring的FactoryBean接口开发的适配器，可以很便捷的通过spring配置设置repository工厂；而RepositoryFactorySupport的作用是，创建一个给定repository接口实例的工厂bean。创建一个实现配置的repository接口的代理，并应用一个advice将控制交给QueryExecuterMethodInterceptor。

我们再进一步看一下我们在Repository接口中定义的查询方法是怎么被识别的，这个问题可以在拦截器中找到答案。我们还记得QueryExecutorMethodInterceptor这个拦截器吗？这个是作为advice加入到proxyFactory里去的。它是RepositoryFactorySupport的内部类，里面的内容并不多。

获取lookupStrategy(查找策略)

方法的查询策略有三种:

  1. Create: 通过解析规范命名的方法名来创建查询, 此时即使有@Query或@NameQuery也会忽略. 如果方法名不符合规则，启动的时候会报异常
  2. USE_DECLARED_QUERY: 也就是使用注解方式声明式创建, 启动时就会去尝试找声明的查询, 也就是使用@Query定义的语句来执行查询, 没有则找@NameQuery的, 再没有就异常.
  3. CREATE_IF_NOT_FOUND: 先找@Query或@NameQuery定义的语句来查询, 如果没有就通过解析方法名来创建查询.

其中, 第三种方式CREATE_IF_NOT_FOUND相当于第一种和第二种的结合版. 也是默认的配置策略.

QueryLookupStrategy 是策略的定义接口，JpaQueryLookupStrategy 是具体策略的实现类

我们关心通过方法名来推导查询，所以直接看CreateQueryLookupStrategy

于是通过方法名推导的query最后会转变为一个PartTreeJpaQuery，它继承AbstractJpaQuery并且实现了RepositoryQuery。AbstractJpaQuery的层次结构如下：

  1. SimpleJpaQuery
    方法头上@Query注解的nativeQuery属性缺省值为false，也就是使用JPQL，此时会创建SimpleJpaQuery实例，并通过两个StringQuery类实例分别持有query jpql语句和根据query jpql计算拼接出来的countQuery jpql语句；

  2. NativeJpaQuery
    方法头上@Query注解的nativeQuery属性如果显式的设置为nativeQuery=true，也就是使用原生SQL，此时就会创建NativeJpaQuery实例；

  3. PartTreeJpaQuery
    方法头上未进行@Query注解，将使用spring-data-jpa独创的方法名识别的方式进行sql语句拼接，此时在spring-data-jpa内部就会创建一个PartTreeJpaQuery实例；

  4. NamedQuery
    使用javax.persistence.NamedQuery注解访问数据库的形式，此时在spring-data-jpa内部就会根据此注解选择创建一个NamedQuery实例；

  5. StoredProcedureJpaQuery
    顾名思义，在Repository接口的方法头上使用org.springframework.data.jpa.repository.query.Procedure注解，也就是调用存储过程的方式访问数据库，此时在spring-data-jpa内部就会根据@Procedure注解而选择创建一个StoredProcedureJpaQuery实例。

总结

lookupStrategy会根据不同的策略将Repository接口里的方法转化为RepositoryQuery也就是AbstractJpaQuery的某种子类，然后保存在queries这个Map<Method, RepositoryQuery>结构。当运行时调用到接口实例的某个方法的时候，由于动态代理的作用，会被QueryExecutorMethodInterceptor拦截，通过调用的方法名找到对应的RepositoryQuery执行execute方法。

至此，为什么只通过定义接口就实现了数据的CRUD的magic被我们揭开了，这都多亏了spring-data-jpa的功劳。
