ConcurrentHashMap源码分析

发表于 2020-04-23 | 分类于 java

ConcurrentHashMap是并发里重要的数据结构，这篇文章结合源码剖析一下为什么ConcurrentHashMap效率足够高。

get

说到map我们肯定就会想到一些常用的用法比如get, put和containsKey等等，我们先从比较简单的get方法来一窥ConcurrentHashMap的究竟

可以看到里面定义了一些局部变量，其中有个Node类型我们还不知道是做什么的，我先把Node的定义贴在下面

从注释可以看到Node实现了Map.Entry，但是它不是对外暴露的用户可修改的Map.Entry，Node里的field有hash，key，val，指向下一个node的next。而在ConcurrentHashMap有两个字段是Node类型，分别是Table和nextTable

table是一个数组，从注释上看出是个列表，并且大小总是2的指数倍，并且是lazy初始化。而nextTable只有在resizing的时候才不为空。

从这里我们大概就能了解ConcurrentHashMap里的构造，首先是有一个Node列表保存的是一级入口，然后在某个entry里面会用链表对于映射到同一个hash的元素通过链表串联。

再看回get方法，首先通过spread算出key的hash值，然后再table列表里找到该hash对应的node节点，然后遍历该node节点所在的链，找到与要查找的key相同key的元素，返回对应value。

乍一看这里面没有并发控制，实际上这个并发控制发生在tabAt

在通过tabAt找到对应的node时候用volatile保证了多线程的可见性。

sizeCtrl用来控制table的初始化和扩容，当sizeCtrl为负数的时候表明table在初始化或者resizing。-1表示初始化，-n表示n-1个活跃线程在负责resizing。当table为null的时候表示表的初始大小，当初始化之后这个值表示需要resizing所需要的元素个数。

resizing

在putVal的最后一步是addCount，如果发现此时table数组太小并且没有在resizing，发起一个transfer，如果已经在resizing，那么当前线程会帮助进行transfer。

根据CPU和当前容量算出每次扩容该分配的区域大小,最小为16,表示为stride.

若过渡数组nextTab未初始化,则先初始化数组.并使用transferIndex记录下旧数组长度,作为扩容长度.

以上扩容需要的数据准备完全开始具体的扩容操作:

在一个while循环中获取本次扩容包含的桶的范围,即[transferIndex,transferIndex-stride]的范围,i表示当前扩容的桶的下标.

三个判断,四段代码分别完成不同情况下的操作

  1. i数值异常,< 0 || >= n || + n > nextn,表示扩容已完成,且在while循环中没有分配的扩容任务.

    • 如果此时finishing参数为true表示整体扩容完成,且完成结束前的检查.

    • 如果finishing为false,则CAS替换sizeCtl为sizeCtl-1,表示一个线程完成扩容任务并需要退出.

      替换成功之后还会检查sc是否等于addCount进来时的值,不相等就直接return,表示还有线程未完成扩容任务.

  2. i对应的桶为空,直接使用ForwardingNode填充头节点,表示此处正在扩容.并设advance为true

  3. 如果检查到节点hash为Moved表示当前节点为ForwardingNode,advance为true.

  4. 排除了上面三种情况,就是对应的桶的迁移工作,和HashMap有点像.结束后设置advance为true

helpTransfer

在任何更新table的时候如果发现桶是moved状态就会调用helpTransfer，当前线程帮助进行扩容。
